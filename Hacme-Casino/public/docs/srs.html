Introduction

HacmeCasino is an online gambling site with multiple introduced security vulnerabilites. It is based on the Ruby on Rails framework and features AJAX support. It is part of the HacmeInc application portfolio from Foundstone.

For more information, see the project proposal.

Use Cases

The main functionality of HacmeCasino should be to allow multiple users to partcipate in an online gambling game, either against real or automated opponents. Rather than go into the specifics of how the gambling room will work, we will say that generally it should look like other gambling sites, like PartyPoker. The use case is that the user goes to the website and signs in, and is allowed to choose a "room", where they can play gambling, until they decide to leave. Their balance, user name, etc, should be persisted.

Also, users should be able to transfer money to HacmeBank. The use case would be that the user registers their HacmeBank account (not quite sure how), and then can place money in to and took money out of the account for use in HacmeCasino.

Let's not forget this is a Hacme app! It needs to have security vulnerabilites. Please see the security vulnerability ideas page for inspirations.

Functional Requirements

The functional requirements for this are as follows:

  • Login and Account Management: The login functionality should be simple, but allow users to be register and be tied to an account which contains the necessary information (balance, name, avatar, whatever).
  • Room Selection: All of the available gambling rooms should be listed here. This should include how many are in their, what the name is, what the game is, etc. This may be an extension point for HacmeCasino
  • Gambling Rooms: This is the big one, where most of the AJAX and security vulnerabilties will sit. Again, the functional requiements are similar to PokerRoom.com application. Generally, you need to be able to see the cards being dealt, place bets, chat (perhaps), and leave the room. Steal, ahem... borrow the sequence of operations from PokerRoom. A big function requirement here (perhaps it should be listed separately) is the possiblity of using automated players (so that a single user can use the app).
  • Balance Transfer: Users should be able to transfer funds to HacmeBank. They would have to be able register their account somehow, and the just transfer balances back and forth.
  • Security Vulnerabilites: These will be ratified, and then the functional requirements will be defined. See security vulnerability ideas page.

Non-Functional Requirements

What are the usability requirements?

The site should look slick, with all sorts of AJAX coolness and eye candy. Think of a casino...they don't have drywall and shop lights.

The site should emulate PokerRoom, PartyPoker, etc, so users immediately "get" how to play. Usability concerns should be tested on beginner and advanced poker players.

Site should fail gracefully, if at all.
