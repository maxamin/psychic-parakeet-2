Glossary
Process impact: The statement of user needs documents and explains the actual desires of stakeholders in roughly their own words. What they desire is never exactly what the product provides. Documenting user needs here, independently from the SRS, helps to keep the SRS precise and makes the tasks of verification and validation more effective. This document is not an informal draft of the SRS, it is different document with a complementary purpose.

Agreed Goals

This project should share the same goals as the HacmeInc applications, which are understood by Foundstone. Additionally, this project should be a "cool" app that demonstrates web 2.0 technologies and is a buzzworth addition to the HacmeInc portfolio.

Environment

What is the system's business environment?
The end user will be an advanced developer, hobbyist, and security knowledgable person.For example, this may be used in a Building Secure Software course. This application will not be used to deal with real-world commercial data.
What is the system's physical environment?
The end user will have a server to install this on. They will want to make this application available to other users, to play.
What is the system's technology environment (hardware and software)?
Who knows? Probably windows. Hopefully similar to development machines :)

Stakeholders / Actors

Foundstone Users
Foundstone users want this application to be fun to demonstrate and representative of their expertise in security.

Key needs:

  • "Cool" look and feel
  • Stable, usable
  • Good security vulnerabilties
Other Users
Other users may want to use this application. Their main concerns are ease of installation and utility of material

Key needs:

  • Easy to install
  • Good training material
